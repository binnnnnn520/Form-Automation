# 问卷自动化本地 Web MVP 设计

日期：2026-05-05

## 1. 产品目标

本产品用于减少用户在公司、学校等场景中反复填写低价值表格和网页问卷的时间成本。第一版目标不是覆盖所有表格系统，而是验证一个核心体验：

用户导入公开网页问卷链接或二维码后，系统读取本地个人资料，调用用户自配的大模型 API，自动把问卷填写到浏览器页面里，并停在提交前由用户确认。

## 2. 目标用户

目标用户是愿意用少量金钱换取大量时间的人，典型场景包括：

- 公司群中的例行登记、满意度调查、活动报名、信息收集。
- 学校群中的问卷、报名表、课程反馈、活动统计。
- 其他公开网页问卷中重复、低价值、非关键决策的信息填写。

## 3. MVP 范围

第一版载体为本地 Web 工具：

- 用户在本机启动服务，通过浏览器访问本地控制台。
- 后端用 Playwright 启动受控浏览器打开问卷。
- 个人资料、模型配置和任务记录保存在本机。
- 大模型调用由用户自己配置 API Key、Base URL 和模型名。

第一版优先支持公开网页问卷，例如问卷星、腾讯问卷、金数据、飞书表单等无需登录、无需验证码、无需复杂人机验证的链接。

## 4. 非目标

MVP 不做以下能力：

- 不自动提交问卷。
- 不处理需要登录、验证码、人机验证、强风控的页面。
- 不做云端账号体系和云端个人资料库。
- 不内置统一模型额度和付费计费系统。
- 不优先处理在线表格逐格填写，例如飞书表格、腾讯文档、金山表格。
- 不承诺覆盖所有动态组件、复杂 iframe 或深度定制问卷页面。

## 5. 用户流程

1. 用户打开本地 Web 控制台。
2. 用户粘贴问卷链接，或上传二维码图片解析链接。
3. 用户维护或选择本地个人资料，例如姓名、学号、工号、部门、住址。
4. 用户配置模型信息，包括 Base URL、API Key、模型名。
5. 用户点击开始分析。
6. 后端用 Playwright 打开问卷页面并抽取题目结构。
7. 系统把题目、选项、必填状态、个人资料传给模型，要求模型返回结构化答案。
8. 系统校验模型答案，并把可安全映射的答案填入页面。
9. 不确定项、无法识别项和敏感风险项显示为需人工处理。
10. 填写完成后，受控浏览器停在提交前，用户自行检查并点击提交。

## 6. 建议技术栈

MVP 使用一套 TypeScript 技术栈，减少前后端模型和类型转换成本：

- 前端：React + Vite + TypeScript。
- 本地后端：Node.js + TypeScript。
- 浏览器自动化：Playwright。
- 本地存储：项目数据目录下的 JSON 文件。
- 二维码解析：前端上传图片后在本地解析，解析结果只作为问卷链接输入。
- 模型接口：OpenAI-compatible `/v1/chat/completions`。

MVP 不引入数据库。后续如果任务历史、统计和多资料档案变复杂，再迁移到 SQLite。

## 7. UI 方向

UI 使用 `frontend-design` 指导，整体参考 Claude 官网的安静、克制、可信气质，但不复制 Anthropic 或 Claude 的品牌标识、文案和专有视觉资产。

视觉方向：

- 暖白或米白背景。
- 深墨色正文。
- 克制边框和柔和圆角。
- 低饱和强调色。
- 绿色或蓝绿色表示已处理。
- 琥珀色表示需要确认。
- 红色只用于错误。

产品形态不是营销落地页，而是直接进入本地自动化控制台。界面应像任务控制室：路径短、状态清楚、隐私边界明确。

核心视图：

- 任务导入：粘贴链接、上传二维码、开始分析。
- 个人资料库：维护姓名、学号、工号、地址、部门等本地字段。
- 模型配置：维护 Base URL、API Key、模型名，并测试连接。
- 填写任务台：展示题目解析、模型答案、填写进度和不确定项。
- 确认页：展示受控浏览器已填好的问卷，等待用户自行提交。

## 8. 架构

```text
用户本地 Web 控制台
  - 粘贴问卷链接 / 上传二维码
  - 编辑本地个人资料
  - 配置 API Key、Base URL、模型名
        |
        v
本地后端服务
  - 用 Playwright 打开公开问卷
  - 抽取问题、选项、输入框和必填状态
  - 组合个人资料和问卷上下文，调用模型生成答案
        |
        v
受控浏览器窗口
  - 自动填入答案
  - 标注不确定项
  - 停在提交前，由用户检查并手动提交
```

## 9. 核心模块

### 9.1 Console UI

本地 Web 控制台，负责用户操作入口：

- 导入问卷链接或二维码。
- 编辑个人资料。
- 配置模型。
- 展示任务状态、问题列表、答案、风险项和填写进度。

### 9.2 Profile Store

本地 JSON 文件保存个人资料和模型配置。MVP 不接入云端资料库，不使用数据库。

保存内容包括：

- 个人资料字段：姓名、学号、工号、部门、地址等。
- 模型配置：Base URL、模型名、API Key。
- 可选任务记录：问卷链接、开始时间、完成状态、错误原因。

API Key 不上传云端。MVP 先保存在本地配置文件中，并在 UI 中明确提示这是本机保存。后续产品化时再接入系统级密钥存储或加密。

### 9.3 Questionnaire Runner

基于 Playwright 的问卷运行器：

- 打开公开问卷链接。
- 等待页面稳定。
- 识别题目文本、控件类型、选项、输入框、必填状态。
- 检测登录页、验证码、人机验证等超出范围的页面。

### 9.4 Model Adapter

统一大模型调用层，优先兼容 OpenAI-compatible Chat Completions API。

用户可配置：

- Base URL。
- API Key。
- 模型名，例如 Kimi、DeepSeek、Qwen 或 GPT 中转模型。

模型输出必须是结构化 JSON，包含题目标识、答案、置信度和原因。

### 9.5 Fill Engine

填写引擎负责把模型答案映射回页面控件：

- 文本题填入输入框。
- 单选题匹配选项并点击。
- 多选题匹配多个选项。
- 下拉题选择匹配项。
- 无法安全匹配的答案不填写，标为需人工处理。

## 10. 错误处理和安全边界

MVP 使用保守策略：遇到不确定情况就停止或交给用户。

规则：

- 链接无法访问：提示网络、超时或链接错误。
- 登录、验证码、人机验证：停止自动填写，提示超出 MVP 范围。
- 题目解析不完整：已识别题目继续处理，未识别题目标为需人工处理。
- 模型答案不匹配选项：不填写，标为需确认。
- 敏感或主观风险题：默认不自动作答，提示用户自行确认。
- 永不自动提交：填写完成后停在提交前。

敏感或主观风险题包括政治态度、健康隐私、法律承诺、财务授权、重要声明等。

## 11. 测试范围

MVP 需要覆盖以下验证：

- 链接导入。
- 二维码解析。
- 本地资料保存和读取。
- 模型配置保存。
- 模型连接测试。
- 示例 HTML 问卷题目抽取。
- 文本框、单选、多选、下拉框填写。
- 模型答案 JSON 校验。
- 无法访问链接的失败状态。
- 登录页、验证码、人机验证的停止状态。
- 模型返回非法格式的失败状态。
- 端到端 smoke：导入测试问卷链接，生成答案，填好页面，停在提交前。

## 12. 后续扩展

验证用户付费意愿后，再考虑：

- 内置模型额度和付费计费。
- 云端账号和跨设备个人资料。
- 浏览器插件形态。
- Electron 桌面应用打包。
- 在线表格逐格填写。
- 更多问卷平台适配。
- 更强的页面控件识别和人工接管能力。

## 13. 开放问题

以下问题不阻塞 MVP，但会影响后续产品化：

- API Key 是否需要在第一版就接入系统密钥链。
- 是否需要任务历史和可复用问卷模板。
- 是否需要为不同场景建立多套个人资料，例如公司、学校、家庭。
- 是否需要对敏感问题增加更细的规则配置。
